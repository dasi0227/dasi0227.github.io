1. 什么是好的关系模式？

考虑教师-系别关系模式in_dept(id,salary,dept_name,budget)，存在两个缺点

  • salary和budget都被重复存储：现实中只要知道了id就知道了salary，只要知道了dept_name就知道了budget
  • 无法添加一个没有老师的新系，除非允许使用空值

分解（decomposition）：将关系模式R分解为两个或多个关系模式R1和R2

  • 有损分解（lossy）：将R1和R2自然连接得到关系R的超集
  • 无损分解（lossless）：将R1和R2自然连接得到关系R

图片

信息是有价值的数据，数据/元组再多但如果是冗余的或者没有意义的，从实际上讲是拥有更少的信息

规范化（normalization）：生成一组关系模式，避免信息冗余，确保数据的完整性和一致性，并且能够高效地进行查询和维护

  1. 验证范式：确定一个给定的关系模式是否满足范式，即是否是良构的
  2. 无损分解：如果不满足，使用算法将关系模式无损分解为许多较小的关系模式

2. 函数依赖（Functional Dependency）

定义：在关系模式r中，对于任意两个元组t1,t2，如果t1和t2在属性集X的值相同，则它们在属性集Y的值也相同，就称Y函数依赖于X，记作：

IF ∀t1,t2∈r,(t1[X]=t2[X])  ⟹  (t1[Y]=t2[Y]) THEN X→Y\text{IF }\forall t_1, t_2 \in r, (t_1[X] = t_2[X]) \implies (t_1[Y] = t_2[Y])\text{ THEN }X \rightarrow YIF ∀t1​,t2​∈r,(t1​[X]=t2​[X])⟹(t1​[Y]=t2​[Y]) THEN X→Y

分类

  • 完全依赖：X->Y，且X没有任何真子集Z也满足Z->Y
  • 部分依赖：X->Y，且X存在一个真子集Z也满足Z->Y
  • 直接依赖：X->Y，且不存在Z使得X->Z->Y
  • 传递依赖：X->Y，且存在Z使得X->Z->Y

3. 范式（Normal Form,NF）

范式：顾名思义，就是规范的关系模式，是设计关系模式的追求目标。范式越高阶，冗余度就越低，设计越好，并且高阶范式一定满足低阶范式

3.1 码

码/超码：唯一标识元组的属性集（可以包含冗余属性）

候选码：唯一标识元组的最小属性集（不包含冗余属性）

主码：由用户选择的其中一个候选码

超码也可以不包含冗余属性，所以超码一定是候选码，候选码是最小的超码
主码是人为选择的，不是数据本身具有的性质，所选的主码一定是候选码，也一定是超码

外码：某属性集是另一个关系的主码

主属性：出现在任一候选码的属性

非主属性：没有出现在任一候选码的属性

3.2 第一范式（1NF）

定义：属性是原子的，也就是说属性不可以分成多个子属性，也不能存在多值属性

存在问题：关系数据库管理系统无法做出非原子属性的表

图片

3.3 第二范式（2NF）

定义：在满足第一范式的基础上，所有非主属性必须完全依赖于候选码，也就是说候选码的真子集不能推出非主属性
（消除部分依赖）

数学表达：

∀A∈R (非主属性),∀K (候选码):¬∃K′⊂K 使得 A→K′\forall A \in R \text{ (非主属性)}, \forall K \text{ (候选码)}: \neg \exists K' \subset K \text{ 使得 } A \rightarrow K'∀A∈R (非主属性),∀K (候选码):¬∃K′⊂K 使得 A→K′

假设存在关系

学生课程教材老师教室时间
达斯数据库数据库系统概念小帅10114:30

易知候选码是（学生，课程），但是有函数依赖：课程->教材，因此该关系模式不满足第二范式，存在以下问题：

  • 插入：如果要新加一门课程，但是还没有人选修，而学生又是主属性不能为空，造成异常；如果要给该课程新加一门教材，则不得不给每个选修该课程的学生都增添一个元组，造成相当大的空间开销
  • 删除：课程结课了，删除全部学生选修该课程的记录，但是如果之后要查找该课程对应的教材就再也找不到了
  • 更新：如果要修改教材名称，不得不对全部选修该课的元组进行修改，造成相当大的时间开销

3.4 第三范式（3NF）

定义：在满足第二范式的基础上，所有非主属性都直接依赖于候选码，也就是说不存在非主属性和候选码之间不存在中间依赖（消除传递依赖）

数学表达：

∀A∈R (非主属性),∀K (候选码):¬∃B (非主属性):A→B 且 B→K\forall A \in R \text{ (非主属性)}, \forall K \text{ (候选码)}: \neg \exists B \text{ (非主属性)}: A \rightarrow B \text{ 且 } B \rightarrow K∀A∈R (非主属性),∀K (候选码):¬∃B (非主属性):A→B 且 B→K

假设存在关系

学生课程老师职位教室时间
达斯数据库小帅系主任10114:30

易知候选码是（学生，课程），但是有函数依赖：（学生，课程）->老师->职位，因此该关系模式不满足第三范式，存在以下问题：

  • 插入：来了一个代课老师，该老师没有职位，职位不得不置空
  • 删除：某个老师只上一门课，该门课结课之后删除对应记录，则再也找不到该老师的职位
  • 更新：老师升职了，不得不对全部包含该老师的元组进行修改，造成相当大的时间开销

3.5 巴斯-科德范式（BCNF）

定义：在满足第三范式的基础上，主属性不依赖于主属性，也就是说每一个非平凡的函数依赖X->Y中X必须是超码

数学表达：

∀X→Y∈F,如果 Y⊈X,则 X 是超码\forall X \rightarrow Y \in F, \text{如果 } Y \nsubseteq X, \text{则 } X \text{ 是超码}∀X→Y∈F,如果 Y⊈X,则 X 是超码

假设有如下关系（仓库ID，物品ID，管理员ID，物品数量），其中有条件：一个管理员只负责一个仓库，一个仓库只有一个管理员，一个仓库可以有多个物品，一个物品可以位于多个仓库

易得候选码有（仓库ID，物品ID）和（管理员ID，物品ID），也就是说非主属性只有物品数量，同时物品数量完全依赖且直接依赖于候选码，所以该模式满足3NF，但是由于存在仓库ID->管理员ID，管理员ID->仓库ID，所以该模式不满足BCNF，存在以下问题

  • 插入：无法做到插入一个新仓库但不分配管理员，或者插入一个新管理员但不分配仓库
  • 删除：清空仓库的物品，则会删除仓库ID和管理员ID
  • 更新：如果要更新管理员，需要对全部元组进行更新，造成相当大的时间开销

应该改为：（仓库ID，管理员ID），（仓库ID，物品ID，物品数量）

3.6 第四范式（4NF）

多值依赖（Multivalued Dependency,MVD）：对于关系R(A,B,C)，如果A的值可以决定B的多个值，并且B和其他属性之间没有依赖关系，那么就称B在给定的A上存在多值依赖，记作A→→BA \rightarrow\rightarrow BA→→B

定义：在满足BC范式的基础上，不存在非平凡的多值依赖X->->Y，也就是消除了属性之间的多对多关系

假设存在如下表：对于每个 StudentID，可以有多个 Course 和多个 Hobby，意味着一个学生可以选修多门课程，一个学生也可以有多个爱好，并且Course和Hobby显然没有任何关系，所以我们说 StudentID 多值依赖于 Course 和 Hobby

4. 函数依赖理论

4.1 阿姆斯特朗公理

三个公理：

  • 自反律：如果Y是X的子集，那么X->Y成立
  • 增补律：如果X->Y，那么对于任意属性集Z，XZ->YZ成立
  • 传递律：如果X->Y和Y->Z都成立，那么X->Z成立
    三个引理：
  • 合并律：如果X->Y成立且X->Z成立，则X->YZ成立
  • 分解律：如果X->YZ成立，则X->Y成立且X->Z成立
  • 伪传递律：如果X->Y成立且ZY->W成立，这XZ->W成立

4.2 函数依赖集的闭包

F的闭包F+是被F逻辑蕴涵的所有函数依赖的集合

4.3 属性集的闭包

函数决定：如果F中存在函数依赖f:X->Y，则称属性集Y被属性集X函数确定

属性集X的闭包是通过F被X所函数决定的所有属性的集合

超码的闭包是整个属性集

4.4 正则覆盖

无关属性（extraneous）：考虑一个函数依赖集F中以及F中的一个函数依赖f，如果删除f中的某个属性X，不改变函数依赖集F的闭包，则称这个属性是无关属性

删除函数依赖f的左侧属性，f变为更强的依赖f’，F变为更强的F’，判断是否有F->F’

F = {AB->C,A->D,D->C}，删除AB->C的B之后变为F’ = {A->C,A->D,D->C}，A->D,D->C通过传递律可以推出A->C，所以有F->F’，所以B是AB->C的无关属性

删除函数依赖f的右侧属性，f变为更弱的依赖f’，F变为更弱的F’，判断是否有F’->F

F = {AB->CD,A->C}，删除AB->CD的C之后变为F’ = {AB->D,A->C}，A->C通过增补律可以推出AB->C，AB->C,AB->D通过合并律可以推出AB->CD，所以有F’->F，所以C是AB->CD的无关属性

正则覆盖（Canonical Cover）：是能保持相同闭包的最小函数依赖集Fc，满足：

  • 每个函数依赖都不可以移除
  • 每个函数依赖都不存在无关属性
  • 每个函数依赖的左侧都是唯一的

考虑F = {A->BC,B->C,A->B,AB->C}

  1. A->BC和A->B可以运用合并律变为A->BC
  2. AB->C中的A是无关的，因为已经存在B->C，所以删除A变为B->C
  3. A->BC中的C是无关的，因为A->B,B->C运用传递律可以得到A->C，而A->C和A->B运用合并律可以得到A->BC，所以删除C变为A->B
  4. 最后得到正则覆盖为Fc = {A->B,B->C}

4.5 保持依赖

保持依赖（Dependency Preservation）：将一个关系模式分解为多个子关系模式后，确保原始关系中的所有函数依赖在新分解后的关系中仍然有效且可被表示

F对Ri的限定Fi：假设R的一个分解是R1,R2,…,Rn，Fi是F+中只包含Ri属性的所有函数依赖的集合，令所有Fi构成F’

  1. 闭包测试：F’的闭包F’+是否和F的闭包F+相同
  1. 函数依赖测试：如果所有的函数依赖f都在至少一个子关系Ri中被保持

5. 分解

令关系模式为R，函数依赖集为F，分解后的关系模式的集合是R’

5.1 3NF分解

关系模式R(A,B,C,D,E,G)，函数依赖集F={B->G,CE->B,C->A,CE->G,B->D,C->D}

  1. 正则覆盖为{B->DG,CE->B,C->AD}
  2. R1 = (B,D,G), R2 = (C,E,B), R3 = (C,A,D)
  3. 候选码是CE，其中(C,E,B)包含CE
  4. R1,R2,R3之间没有包含关系

综上R’ = {(B,D,G),(C,E,B),(C,A,D)}

5.2 BCNF分解

关系模式R(A,B,C,D,E,G)，函数依赖集F={A->B,B->C,AD->G,D->E}

  1. R的候选键为AD，A->B不满足BCNF，令R1 = AB，R2 = ACDEG
  2. R1的候选键为A，保持的函数依赖为A->B，满足BCNF
  3. R2的候选键为AD，其中A->C不满足BCNF，令R3 = AC，R4 = ADEG
  4. R3的候选键为A，保持的函数依赖为A->C，满足BCNF
  5. R4的候选键为AD，其中D->E不满足BCNF，令R5 = DE，R6 = ADG
  6. R5的候选键为D，保持的函数依赖为D->E，满足BCNF
  7. R6的候选键为AD，保持的函数依赖为AD->G，满足BCNF

综上，R分解为{(A,B),(A,C),(D,E),(A,D,G)}

5.3 4NF分解

关系模式R(A,B,C,D)，F={A->->B,A->C,C->->D}

  1. A->->B违反4NF，令R1=(A,B)，R2=(A,C,D)
  2. C->->D违反4NF，令R3=(C,D)，R4=(A,C)

综上，R分解为{(A,B),(A,C),(C,D)}
