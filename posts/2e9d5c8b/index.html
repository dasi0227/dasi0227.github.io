查询处理

发表于2024-12-08|笔记
|字数总计:3.5k|阅读时长:12分钟|阅读量:

1. 查询处理

处理流程

  1. 查询分析：检查查询语句的拼写、语法、权限、安全性等
  2. 查询翻译：将查询语句转换为内部的表示形式，通常是关系代数表达式
  3. 查询优化：构造具有最小查询执行代价的查询执行计划（原语操作序列）
  4. 查询执行：代码生成器接受优化器的查询执行计划并生成内部执行代码，然后由执行引擎执行代码并返回结果

查询原语：数据库引擎用来执行查询操作的基本运算步骤

  • 选择（selection）：根据选择条件筛选元组
  • 连接（join）：根据连接条件合并两个元组
  • 排序（sort）：根据排序键排序元组
  • 投影（projection）：根据投影条件筛选属性

2. 查询代价

影响因素

  • 磁盘访问：读取或写入数据时进行磁盘 I/O 操作
  • CPU资源：查询进程是否需要排队等待
  • 网络通信：对于分布式数据库系统，需要考虑网络延迟

代价度量

  • 响应时间：查询从发起到返回结果所经历的时间
  • 资源消耗总量：执行查询时消耗的资源总量

优化器通常努力去将查询计划的资源消耗总量降到最低，而不是努力将响应时间降到最低，这是因为响应时间依赖于缓存和磁盘物理布局，而数据库无法预先知道这些信息，也就无法估计响应时间

Cost=b×tT+S×tSCost = b \times t_T + S \times t_SCost=b×tT​+S×tS​

  • b×tTb \times t_Tb×tT​：读取或写入磁盘的b个块所需的总传输时间
  • S×tSS \times t_SS×tS​：定位到磁盘的S个数据块所需的总寻道时间

假设平均传输时延和平均寻道时延是固定的，因此只需要考虑传输次数bbb和寻道次数SSS即可
如果是顺序扫描相邻磁盘块，可以认为寻道时间为0

内存影响

  • 某些数据可能已经位于内存缓冲区中，因此不需要进行磁盘I/O
  • 可以将内存的一部分用于输入/输出缓冲，则一次可以传输多个块，从而减少寻道次数
  • 内存是动态变化的，可能有时候可以一次读取全部数据
  • 最坏情况：初始内存缓冲为空、没有设置输入缓冲区和输出缓冲区，可用内存始终有限

实际系统中进行乐观估计可能会更接近真实性能，但是具有一定风险和不稳定性

3. 选择运算

选择（selection）：筛选行，也就是提取满足特定条件的记录

复杂选择

算法代价分析
A7：使用一个索引的合取选择先利用索引搜索，然后再执行线性搜索
A8：使用组合索引的合取选择传输次数=寻道次数=h+n

索引搜索相较于线性搜索，虽然可以减少传输次数，但是可能会增加寻道次数，由于传输时延小于寻道时延，所以索引搜索更适合大规模数据（传输作为瓶颈），线性搜索更适合小规模数据（寻道作为瓶颈）

4. 排序运算

外排序-归并算法：假设一共有B个数据块，可用内存为M个数据块

  1. 创造归并段：每次从磁盘中读入M个数据块到内存中，然后进行内排序，将排序好的M个数据块作为一整个归并段写回磁盘，因此总共会生成⌈B/M⌉\lceil B/M \rceil⌈B/M⌉个归并段（最后一个归并段可能不满M个数据块）
  2. 归并趟（趟-次-轮）
    1. 每趟为每个归并段分配n个内存数据块作为输入缓冲，并总是保留一个内存数据块作为输出缓冲
    2. 每次可以将⌊M/n⌋−1\lfloor M/n \rfloor - 1⌊M/n⌋−1个归并段合并为一个归并段
    3. 每轮对输入缓冲进行排序，将结果放到输出缓冲，然后补充归并段的块到输入缓冲中
    4. 如果输出缓冲满了，就写入磁盘作为新的归并段的一部分
  3. 一共需要log⌊M/n⌋−1(B/M)log_{\lfloor M/n \rfloor - 1}(B/M)log⌊M/n⌋−1​(B/M)趟归并

传输代价

  • 创造归并段：读取全部数据块进行内排序，然后又写回全部数据块，因此需要2B2B2B次传输
  • 归并趟：每趟都需要读取和写回全部数据块，因此需要B∗2log⌊M/n⌋−1(B/M)B * 2log_{\lfloor M/n \rfloor - 1}(B/M)B∗2log⌊M/n⌋−1​(B/M)次传输

寻道代价

  • 创造归并段：每次读取M个数据块和写入一个归并段都需要一次寻道，因此需要2⌈B/M⌉2\lceil B / M \rceil2⌈B/M⌉次寻道
  • 归并趟：每次读取和写入都需要寻道，但由于缓冲区的存在，只用读取起始数据块即可，所以需要⌈B/n⌉∗(2log⌊M/n⌋−1(B/M)−1)\lceil B/n \rceil * (2log_{\lfloor M/n \rfloor - 1}(B/M) - 1)⌈B/n⌉∗(2log⌊M/n⌋−1​(B/M)−1)次寻道

注意最后一次不需要写回，所以寻道次数需要-1

5. 连接运算：r⋈θsr \bowtie_\theta sr⋈θ​s

5.1 嵌套循环连接

5.1.1 元组嵌套循环连接

算法流程

代价

  • 传输次数=br+nr∗bs\text{传输次数} = b_r + n_r * b_s传输次数=br​+nr​∗bs​：外循环传输brb_rbr​个块，内循环对于r的每个元组都需要传输s的块
  • 寻道次数=br+nr\text{寻道次数} = b_r + n_r寻道次数=br​+nr​：每次内循环都需要找到s的起始块，每次从内循环返回外循环都需要找到r的遍历块

5.1.2 块嵌套循环连接

算法流程

代价

  • 传输次数=br+br∗bs\text{传输次数} = b_r + b_r * b_s传输次数=br​+br​∗bs​：外循环传输brb_rbr​个块，内循环对于r的每个块都需要传输s的块
  • 寻道次数=2∗blockr\text{寻道次数} = 2 * block_r寻道次数=2∗blockr​：从外循环进入内循环寻道一次，从内循环返回外循环寻道一次

区别在于内循环不是遍历关系r的元组，而是遍历关系r的块，显著降低了传输次数和寻道次数

5.1.3 带有缓冲区的块嵌套循环连接

假设可用内存块数为M，分配M-2个块作为外循环关系的缓冲区，剩下两个块作为内循环关系的缓冲区

  • 传输次数=br+⌈br/(M−2)⌉∗bs\text{传输次数} = b_r + \lceil b_r/(M-2) \rceil * b_s传输次数=br​+⌈br​/(M−2)⌉∗bs​：由于外循环缓冲了M-2个块，所以每M-2次从内循环返回外循环才需要重新寻道一次，而一共有⌈br/(M−2)⌉\lceil b_r/(M-2) \rceil⌈br​/(M−2)⌉次从内循环返回
  • 寻道次数=2∗⌈br/(M−2)⌉\text{寻道次数} = 2 * \lceil b_r/(M-2) \rceil寻道次数=2∗⌈br​/(M−2)⌉：同上，每M-2次从外循环进入内循环和从内循环返回外循环才需要寻道一次

5.1.3 索引嵌套循环连接

算法流程

代价：假设B+树的高度为h

  • 连接过程=br(tT+tS)\text{连接过程} = b_r(t_T + t_S)连接过程=br​(tT​+tS​)：只有外关系r需要传输和寻道
  • 选择过程=nr∗cost\text{选择过程} = n_r * cost选择过程=nr​∗cost：对于r的每个元组都需要将连接条件转换为选择条件去选择对应的元组s，cost是索引选择代价

适用于索引选择代价比较小的情况

5.2 归并排序连接

5.2.1 归并排序后连接

算法流程

代价

  • 传输次数=blockr+blocks\text{传输次数} = block_r + block_s传输次数=blockr​+blocks​：按照顺序传输即可
  • 寻道次数=⌈br/bb⌉+⌈bs/bb⌉\text{寻道次数} = \lceil b_r/b_b \rceil + \lceil b_s/b_b \rceil寻道次数=⌈br​/bb​⌉+⌈bs​/bb​⌉：假设分配bbb_bbb​个内存块给每个关系，只有缓冲区的块用完了才需要新的寻道
  • 上述代价只是连接代价，还需要考虑排序代价

归并连接只适用于等值连接和自然连接

5.2.2 归并排序时连接

算法流程：假设归并段的总数小于内存块数，因此可以同一时间对全部归并段进行外排序

代价

  • 传输次数
    • 创造归并段需要读取全部块，然后写回：2(br+bs)2(b_r+b_s)2(br​+bs​)
    • 归并阶段按顺序每个块即可：(br+bs)(b_r+b_s)(br​+bs​)
  • 寻道次数
    • 创造归并段，假设每M个数据块的读取和写回需要一次寻道：2(⌈br/M⌉+⌈bs/M⌉)2(\lceil b_r/M \rceil + \lceil b_s / M \rceil)2(⌈br​/M⌉+⌈bs​/M⌉)
    • 归并阶段，假设为每个归并段分配bbb_bbb​个块作为输入缓冲：⌈br/bb⌉+⌈bs/bb⌉\lceil b_r / b_b \rceil + \lceil b_s / b_b \rceil⌈br​/bb​⌉+⌈bs​/bb​⌉

5.3 哈希连接

原理

  • 构建阶段：使用第一个哈希函数，将数据按照连接键的值分成多个连续的区域
  • 探测阶段：在同一个分区中，使用第二个哈希函数，按照连接键的值实现元组匹配

算法流程

代价

  • 传输次数
    • 构建阶段：读取和写回全部数据块，写回nkn_knk​个分区，一共需要2(br+bs+nk)2(b_r+b_s+n_k)2(br​+bs​+nk​)次传输，
    • 探测阶段：按顺序读取分区即可，所以需要br+bsb_r+b_sbr​+bs​次传输
  • 寻道次数：假设为每个块分配的缓冲区大小为bbb_bbb​块假设可用内存为M块，那么无论是探测阶段还是构建阶段，都需要⌈br/bb⌉+⌈bs/bb⌉\lceil b_r / b_b \rceil + \lceil b_s / b_b \rceil⌈br​/bb​⌉+⌈bs​/bb​⌉次寻道

溢出（overflow）

  • 现象：某个分区的大小远大于内存大小
  • 原因：哈希函数1的选择不好，数据分布很密集
  • 解决：再哈希，更换哈希函数，对大分区使用块嵌套循环连接

6. 执行模式

执行模式指的是如何处理包含多个运算的表达式

6.1 物化（materialized）

定义：中间结果集被完全计算并存储到内存或磁盘中，后续的查询步骤直接从存储的结果集获取数据

代价 = 每个运算的代价 + 将中间结果写回磁盘和从磁盘读取中间结果的代价

双缓冲技术：一个缓冲区用于执行运算，另一个缓冲区用于存储中间结果，从而避免中间结果的I/O操作

6.2 流水线（pipeline）

定义：每一步的计算结果直接传递给下一步进行处理，无需存储到磁盘或缓冲区

局限性：流水线模式要求生成数据流的操作能够即刻产生结果，而不是等待所有输入都接收后再计算

  1. 排序通常需要所有输入数据可用后才能进行有效排序
  2. 哈希连接需要整个构建阶段完成后才能进入探测阶段

驱动方式

  • 需求驱动模式 (Demand-Driven) / 拉模式（Pull）/ 按需模式（Lazy）：每当上游操作需要数据时，下游操作才生成数据
  • 生产驱动模式 (Producer-Driven) / 推模式（Push） / 驱动模式（Eager）：下游操作不断执行，主动将数据传递给上游
